# GitHub 仓库整理建议

这份整理按“一个仓库只放一个清晰用途”的原则来分，方便以后维护。

## 建议的 3 个项目分组

### 1. `daily-ai-brief`

用途：每天自动抓取 AI 论文、新闻和官方博客，筛选后生成本地 HTML / Word 简报，后续接入飞书自动发送。

当前状态：已经有可运行代码，适合优先上传。

建议上传内容：

- `src/`：核心程序。
- `scripts/`：导出 HTML / Word 的辅助脚本。
- `config.example.json`：示例配置。
- `.env.example`：飞书、API Key 等私密配置模板。
- `README.md`：项目说明。
- `docs/`：产品和仓库整理文档。

不建议上传内容：

- `data/`：每天下载的 PDF 和网页资料。
- `reports/`：每天生成的中间 Markdown 报告。
- `outputs/`：每天生成的 HTML / Word 成品。
- `.env`：任何账号、密钥、飞书凭证。

### 2. `feishu-automation-playground`

用途：专门测试飞书 API，包括创建云文档、写入正文、发送消息、权限配置。

为什么单独分：飞书接入会涉及开放平台权限、机器人消息、文档块写入等问题，和报告内容生成是两类风险。拆开以后，调飞书不会影响日报生成。

当前状态：建议等飞书 API 凭证准备好后再建。

### 3. `ai-research-workbench`

用途：存放你后续可能想做的研究工具，比如文章逻辑图生成、网页截图留档、原文证据抽取、文章质量评分等。

为什么单独分：这些能力未来可以被日报项目调用，也可以服务别的项目。先不急着建，等功能变多后再拆出来。

## 现在最推荐的动作

先把 `daily-ai-brief` 作为第一个 GitHub 仓库上传。它已经有完整雏形，也最贴近你现在真正要用的自动化流程。

GitHub 侧目前我没有看到已授权仓库，所以还不能直接上传。你可以任选一种方式继续：

- 给我一个已有仓库链接，我把这个项目放进去。
- 在 GitHub 新建一个空仓库，建议名字叫 `daily-ai-brief`，然后把链接发我。
- 把 GitHub App 授权到你的目标仓库后，我再继续上传。
